# retold-monorepo-manager — Refactor Plan

Refactor `retold/source` (the `retold-manager` web tool) into a standalone, generic,
config-driven monorepo manager packaged like `ultravisor`: one shared core driving both a
CLI and an Orator web server, dockerizable, with docs/tests/release trimmings. Strip all
fork / pull-request / multi-state machinery. Keep and massively expand the
dependency-graph ripple engine. Make everything run from a plain `Modules-Manifest.json`
so it works on any monorepo (subfolders of modules, a docs folder, spread-out
package.jsons, services in odd folders).

Target location: `retold/modules/apps/retold-monorepo-manager/`

---

## 0. Findings that shape this plan

These come from reading the current code end-to-end. They change the shape of the work,
so they lead.

1. **The CLI is greenfield.** The old `blessed` TUI was retired; today the tool is
   **web-only** (`retold/source/retold-manager-web.js`, default port 44444; `--terminal`
   /`--tui` are silent no-ops). So "each action needs a clean CLI equivalent" is a
   *new build*, not a port. The upside: we get to design the command grammar cleanly and
   make the web UI call the *same* command map.

2. **The two module lists are tabs, not peers, and the docks already exist.** The
   "sidebar full of modules" is the **Modules** pane — one of three tabs
   (Modules / Files / Search) in `PictView-Manager-Sidebar.js`. The "bottom bar full of
   modules" is the **Modules/scan** tab — one of three tabs (Actions / Log / Modules) in
   `PictView-Manager-LogBar.js`. Both read the same `AppData.Manager.Scan.Results` and the
   same classifier `Manager-Scan-State.js`. The layout is built with
   `pict-section-modal`'s `shell()` + `addPanel()` — a **left `sidebar` panel**
   (`#RM-Sidebar-Content`) and a **bottom `logbar` panel** (`#RM-LogBar-Content`) already
   exist. So "one movable list" = extract the list-with-column-projection into its own view
   and let a button re-point which panel hosts it. We do **not** invent a docking system.

3. **`RippleGraph.js` is ~90% generic already** (752 lines). Graph build, transitive-consumer
   BFS and Kahn topo-sort have zero fork coupling. The only leaks are a hardcoded
   `const GROUP_ORDER = ['Fable','Meadow','Orator','Pict','Utility','Apps']` (line 38,
   even `module.exports`-ed at 751) and retold-named action templates
   (`ncu-retold`, `BringRetoldDepsForward`, `prepare-docs`, a "stay out of
   retold-remote/retold-databeacon" default). **All fork/PR ops live in one place** —
   `_buildFlatPlan` (lines 592–747) emits `merge-upstream / sync-upstream / create-pr /
   approve-pr / merge-pr`; the executor's fork cases are `Api-Ripple.js:464–848`.

4. **"Ecosystem membership" is manifest-membership, not npm scope.** In
   `ManifestLoader` every module name is added to `ecosystemNames` (a `Set`); the
   introspector and ripple graph treat a dependency as in-ecosystem iff its package name
   *is a manifest module name*. Retold's own packages are mostly unscoped (`pict`, `fable`,
   `meadow`). **Any design that redefines "ecosystem" as an `@scope/` prefix silently breaks
   the stale-dep check and the ripple edges.** Keep manifest-membership as the default;
   make npm-scope filtering an *additive* option.

5. **Deleting the fork scripts orphans two features unless we build replacements.**
   `mm all status|update|checkout|install` today shells `modules/*.sh` scripts that were
   *generated by* `manifest-rebuild-shell.js` — which we're deleting. And `deps sync`'s real
   implementation is inseparably fork/quack/push-coupled. Both need small native
   re-implementations (a ProcessRunner fan-out; a version-source aligner), not a "keep".

6. **The introspector's fork logic is ~16% of the file** (~230–250 of 1424 lines), threaded
   through both the sync path (`getGitStatus`) and the async scan (`scanAllModulesAsync`).
   The collapse to origin-only is a real rewrite with a matching client change
   (`Manager-Scan-State.js` action vocabulary), not a one-line delete.

7. **The ultravisor brand step reads retold's central manifest at build time.** The
   `webinterface` prebuild runs `pict-section-theme-brand --manifest ../Retold-Modules-Manifest.json`.
   A truly standalone package must vendor its own brand block and drop that external read,
   or the "generic" tool silently depends on retold's manifest to build.

## 3. The generic `Modules-Manifest.json`

Neutral name, no "Retold". Declare-authoritative (nothing scans at read time); discovery
runs only on the explicit `manifest backfill`/`audit` commands. Two entry shapes tolerated
(rich module vs lightweight example); loader must not assume `GitHub`/`Documentation`/
`RelatedModules` exist.

```jsonc
{
  "SchemaVersion": "2.0",
  "Name": "My Monorepo",
  "Description": "…",

  "RepoRoot": null,          // defaults to the manifest's own directory
  "GitRemote": "origin",     // single-remote model. No upstream/fork.
  "DefaultBranch": "main",   // retires scattered 'master' fallbacks

  // URL synthesis for `manifest backfill` (replaces baked GitHubOrg='fable-retold')
  "Org": "my-org",
  "GitTemplate": "https://github.com/{org}/{name}.git",
  "DocsTemplate": "https://{org}.github.io/{name}/",

  // "ecosystem" = which deps are treated as in-repo for stale-dep + ripple edges.
  // DEFAULT 'manifest' = every module in this file (matches today's semantics).
  // 'scopes' filters by npm name-prefix; 'both' = union. Do NOT default to scopes.
  "EcosystemMembership": { "Mode": "manifest", "Scopes": [] },

  // Ripple: per-node action templates. Fully config-driven. No PR/fork ops exist.
  "Ripple": {
    "GroupOrder": [],        // [] = topo-only tie-break by name; array = declared group precedence (was GROUP_ORDER)
    "ConsumerBump": "patch",
    "ProducerBump": "patch",
    "WaitForIndex": true,    // now a REAL action step in the runner registry
    "ProducerActions": ["preflight-clean-tree","bump-if-needed","publish","wait-for-index","commit-final","push"],
    "ConsumerActions": ["preflight-clean-tree","update-dep","install","test","commit","bump","publish","wait-for-index","commit-final","push"]
  },

  // deps sync needs a version-of-truth (retold assumed a single umbrella package.json)
  "VersionSource": "highest-in-repo",   // | "root-package" | "versions-map:<path>"

  "Docs":    { "Path": "docs", "Engine": "npx quack prepare-docs ./docs" },
  "Logging": { "LogFilePrefix": "Monorepo-Manager-Operations-", "LogDir": ".", "Sink": "file" },
  "Auth":    { "Enabled": false, "Provider": null },
  "WebServer": { "Port": 44444, "Host": "127.0.0.1" },

  "Groups": [
    {
      "Name": "Pict",
      "Description": "MVC tools",
      "Path": "modules/pict",          // group folder root (location anchor)
      "DiskName": "pict",              // only when display name != folder; removes the hardcoded alias table
      "Discover": ["*"],               // glob(s) under Path for backfill/audit (NOT live). '*' = one-level dirs
      "ModuleMarker": "package.json",  // a dir is a module iff it has this file; lets non-npm services count
      "Modules": [
        {
          "Name": "pict-section-form",
          "Path": "modules/pict/pict-section-form",  // entry Path WINS over group convention → services in odd folders
          "Type": "library",                          // open enum: library|service|webapp|app|example|tool
          "Description": "…",
          "GitHub": null,             // string | false(opt-out) | null(synthesize on backfill)
          "Documentation": false,
          "DocsPath": "docs",
          "RelatedModules": [],
          // OPTIONAL — drives the generic ServiceSupervisor + `mm service …`
          "Service": { "Entry": "source/App.js", "Port": 8080, "StartCommand": "npm start" }
        }
      ]
    }

`mm manifest migrate` converts retold's v1 (`{Generated,Description,GitHubOrg,Groups[]}` with
`Forkable`/`Owner`) to v2: drop fork fields, lift `GitHubOrg`→`Org`+`GitTemplate`, seed
`GroupOrder` from the old order if desired, set `EcosystemMembership.Mode="manifest"`.

## 4. Ripple engine — split three ways, then expand

`RippleGraph.js` is already ~90% clean; the fix is structural, not a rewrite. Split into:

**(a) `Manager-Core-DependencyGraph.js` — pure.** Lift verbatim from `RippleGraph.js`:
`build()`, `_readPackageJson`, `_collectEdges`, `getTransitiveConsumers(OfAll)`, `topoSort`,
`invalidate`. Edge rule unchanged: an edge exists iff a package.json dep name is a manifest
module name (via the ecosystem `Set`). **Replace the hardcoded `GROUP_ORDER` (line 38) with an
injected `GroupOrder` option** used only for topo tie-breaking (default: name order). Add pure
query methods that enable the expansion — all computable from the existing indexes:
`impactOf(root)` / `blastRadius(roots)` (transitive-consumer closure + count),
`dependenciesOf`/`dependentsOf`, `subgraph(nodes)`, `findCycles()` (surface the residual-in-degree
set the topo fallback already detects), `toVisualizationJSON()` (nodes colored by group, edges
labeled by section/range). No fs writes, no git, no npm → trivially unit-testable.

**(b) `Manager-Core-PlanComposer.js` — config-driven.** Generalize `buildPlan`'s per-node loop:
given topo order + configured `ProducerActions`/`ConsumerActions` templates, emit an action chain
per node. `update-dep` synthesis (earlier-in-cone dep detection) stays as composer logic (it's
graph-derived). **Delete `_buildFlatPlan`'s fork emissions entirely.** `ncu-retold` → generic `ncu`
with scope from `EcosystemMembership`; `prepare-docs` → optional named step. Plan shape
(`{PlanId,Roots[],Graph{Nodes,Edges},Steps[{Order,Module,Kind,Actions[]}]}`) is already generic — keep it.

**(c) `Manager-Core-RippleRunner.js` — action registry.** Lift `executeRipple`/`runAction`/
`runPublishWithConfirm`/`awaitRegistryCatchUp`/`failRipple`/`finalizeRipple` + the confirm/cancel/
retry lifecycle from `Api-Ripple.js`. Convert the big `switch(Op)` into a **registry**
`Op → async handler(context, entry, action)` so hosts register actions instead of editing a switch.
Ship default handlers for the pure ops: `preflight-clean-tree, update-dep, install, test, commit,
bump, bump-if-needed, publish, wait-for-index, commit-final, push`. **Promote the npm-index wait**
(today inline at `Api-Ripple.js:877–951`) to a first-class `wait-for-index` handler so the config
`WaitForIndex` flag toggles a real step. Delete the five fork cases (`:464–848`) and the
`libGitHubPr` import (`:42–50`). The WS pause/confirm lifecycle is generic — keep it. Add a
`DryRun` mode that walks the plan emitting the same frames without invoking ProcessRunner.

**Expansion (new capabilities, all unlocked by the split):** `mm ripple graph|impact`, partial-graph
& multi-root plans, dry-run preview, a graph-visualization panel in the web `Ripple` view, and
what-if version diffs (reuse `compareSemver` + on-disk version read, no writes).

## 5. The two orphaned features (must-build, not keep)

**`mm all` → `Manager-Core-AllModulesRunner.js`.** Today `all status|update|checkout|install`
shells `modules/Status.sh` etc., which were *generated by* the deleted `manifest-rebuild-shell.js`.
Replace with a native ProcessRunner fan-out over `ManifestLoader` modules: `status` = per-module
`git status` roll-up (reuse the introspector), `update` = `git pull` each, `install` = `npm install`
each, `checkout` = `git clone <GitRemote-derived-url> <AbsolutePath>` for any missing module (this
is genuinely new — nothing clones from the manifest today). Concurrency-capped, streamed through the
broadcaster. *(Optional:* keep a `mm manifest rebuild-scripts` that emits generic per-monorepo shell
scripts for users who want them — off by default.)*

**`deps sync` → `Manager-Core-DepAligner.js`.** The retold script is fork/quack/push-coupled and its
commit half is deleted. Build the reusable nugget fresh: read the declared `VersionSource`
(`highest-in-repo` default | `root-package` | `versions-map:<path>`), rewrite each sub-package's
dep/devDep ranges to match (atomic tmp+rename, preserve indentation), leave changes uncommitted for
review. No push, no docs coupling.

## 6. Web UI — one movable module list

**Extract, don't rebuild.** Pull the module-list-with-column-projection out of Sidebar's Modules
pane and LogBar's Modules tab into a new `PictView-Manager-ModuleList.js` with a
`variant: 'compact' | 'wide'` option and **one** shared filter/sort pipeline (today duplicated across
`Sidebar._buildModulesPane` and `LogBar._buildScanRecord`). Both read the same
`AppData.Manager.Scan.Results` + `Manager-Scan-State.js`, so this is a consolidation, not new logic.

## 8. CLI + the UI↔CLI command map

Grammar: `mm <noun> <verb> [target] [flags]`, built on `pict-service-commandlineutility`. Manifest
auto-discovered by walking up from cwd (git-style) when `--config` is omitted. The **CommandMap** is
the single source of truth: each entry is `{Noun, Verb, CoreMethod|Route, HttpMethod, Transport, Args}`.
The CLI dispatcher and the web client both read it, so a new action is added in one place and the UI
button + CLI verb can't drift. **Each entry carries a `Transport` tag** — `native` (runs in-process,
identical CLI/web), `web-only` (needs a live server/WS op-id), or `mode-divergent` (e.g. the
PreviewHash publish handshake — the CLI recomputes the hash fresh in-process rather than pretending to
share server session state). This is the honest version of "every action has a CLI equivalent."

```
STATUS      mm status [--fetch] [--dirty-only] · mm status published · mm show <m> · mm health · mm web [--port]
LIFECYCLE   mm run <m> install|test|types|build · mm run <m> -- <cmd …> · mm version <m> patch|minor|major|X.Y.Z
            mm publish check <m> · mm publish run <m> [--yes]        (mode-divergent: recompute PreviewHash)
GIT         mm git <m> pull|push|fetch|diff|log|add|commit -m "…"    (origin only — no sync/create-pr)
DEPS/DOCS   mm deps <m> ncu [--apply] [--scope ecosystem|all] · mm deps sync [--limit N]
            mm docs regen <m> · mm docs serve|edit <m> [--stop|--status]
SERVICES    mm service <m> start|stop|status                        (generic supervisor; "services in odd folders")
RIPPLE      mm ripple graph [--root <m>] [--json] · mm ripple impact <m> [--depth N]
            mm ripple plan <m…> [--dry] · mm ripple run <m…> [--yes] · mm ripple resume|cancel <id>
REPO-WIDE   mm all status|update|install|checkout                   (native fan-out, §5)
            mm system npm-cache clean|verify
MANIFEST    mm manifest audit|backfill [--write]|reload|migrate
            mm manifest group add|edit|remove <name> · mm manifest module add|edit|remove|move <name>
FILES/OPS   mm files <m> ls|cat <path> · mm search <q> [--scope …] · mm ops tail|cancel <id> (web-only) · mm log [--tail N]
```

`docs/cli-reference.md` is generated from the CommandMap so it never goes stale.

## 9. Docker, users, logging

**Docker** mirrors ultravisor (two-stage, multi-arch, `npm install`, GHCR on tags). One difference:
the manager operates on a **foreign monorepo mounted at runtime**. Document a bind-mount
(`-v /path/to/monorepo:/monorepo`) + `--config`/`MONOREPO_ROOT`; git in the runtime image; and, for
push, mounted credentials (`~/.gitconfig` + SSH agent, or a token) — flag the security surface. Add a
HEALTHCHECK on `/api/manager/health`, restart-policy `unless-stopped`.

**Users (seam only, not built):** config `Auth:{Enabled:false}`. In `Server-Setup`, one conditional
middleware mount after bodyParser (`orator-authentication` populating `pRequest.UserID`). Register all
routes through a single `wireRoutes(core, wrap)` so a future authorization wrapper decorates every
handler in one place. Handlers read `pRequest.UserID` defensively (undefined today). Reintroduce a
`TopBar-User` view behind this seam later.

**Logging (single chokepoint):** `OperationLogger` already captures every ProcessRunner start/line/
end/error to a daily-rolling transcript. Add an optional `Actor` field (from `pRequest.UserID`) later —
no new call sites, since every mutating op already flows through it. Keep `LogFilePrefix`/`LogDir`/`Sink`
config-driven (point at a mounted volume; `Sink` is the hook for an external store). Follow the Fable
logging convention (inject `this.log`/`this.fable.log`; structured datum as the *second* arg).

**Sever the retold build dependency:** the webinterface prebuild must **not** read retold's central
manifest. Vendor a self-contained `retold.brand` block in `webinterface/package.json` and a
`docs/_brand.json`; drop `--manifest ../Retold-Modules-Manifest.json` from the brand step.
